AI DEPLOYMENT SWEEP — agent rules (condensed, v2 "new deal"). Full: AGENT-MANIFESTO.md

MISSION: for each company in your country's universe file, find REAL AI implementations. Sample not census. Something>Nothing>Poop.

SEARCH (proven over 14 countries — follow exactly):
- Query in the TARGET LANGUAGE (see your country). English-only searching was the single biggest miss — native queries surface primary in-country sources.
- PRIMARY PATH (reliable): WebFetch against  https://html.duckduckgo.com/html/?q=<url-encoded native query>  — bypasses throttling.
- Fallback: python3 ~/.claude/tools/websearch.py "query" (has retry/backoff). CN: also --engine baidu.
- PACING IS MANDATORY: strictly sequential, ~25-40s between queries, wait 60s + retry on 403/empty. NO parallel queries, NO sub-agents, NO monitors, NO polling. Grind + checkpoint per company.

PER COMPANY (walk your universe file top to bottom):
1. search company name + AI (native language).
2. Classify each real hit on EXISTENCE:
   CONFIRMED = company's own disclosure / filing / press release / named deployment
   CLAIMED   = 3rd-party report, no primary source
   SMOKE     = vendor blog / SEO / aggregator / "book a demo"  -> REJECT, do not record
3. TWO AXES (never one blurry "row" — the word "row_assigned" is DEAD):
   vertical   = the company's INDUSTRY (bank -> Financial Services). Controlled list in CLASSIFICATION.md.
   horizontal = what the AI DOES, pick ONE of exactly six:
        Core / Domain  | Customer Support | Software / Code | Sales / Marketing | Back-office / Ops | Security / Risk
        (bank chatbot -> Customer Support; bank fraud model -> Core / Domain)
   raw_sector = the company's original sector label from the universe file (preserve verbatim).
   One deployment = one row.
4. VALUE: a self-claimed number if given, else "none". (independent/audited value = rare; note LOUDLY if found.)
5. TIER: P = primary (filing/company release) | I = indirect (credible news) | S = smoke (reject, don't record).
6. **DATE (NEW — always capture):** the year (YYYY) or YYYY-MM when the deployment was announced/reported,
   taken from the SAME source you cite (press-release date, filing date, article date). Write "missing" ONLY if
   genuinely absent. A dateless SEO page is itself a rejection signal. Prefer dated sources.
7. KEEP the source URL always. No URL = don't record it.
8. Nothing real for a company: record "no confirmed deployment found" — do NOT invent.

OUTPUT — append per company (checkpoint) to /tmp/register_<CC>.md as pipe-table rows.
EXACTLY 11 fields, one clean value per column. The existence token goes in its OWN column —
NEVER append "| CONFIRMED" to the use_case text (that caused 772 drifted rows last time):

company | country | vertical | raw_sector | horizontal | use_case | existence | value_claimed | tier | source_url | date

- country = your 2-letter code, same on every row (do NOT put the company's native name here).
- use_case = plain description, NO trailing tokens, NO pipes inside it.
- Example:
  DBS Group | SG | Financial Services | Banks | Customer Support | AI virtual assistant handling 90% of chat queries | CONFIRMED | 90% of queries automated | P | https://dbs.com/... | 2024

ALSO append qualitative signals (skeptic press, regulator warnings, surprising numbers, documented failures)
to /tmp/findings_<CC>.md: vertical | finding | source_url | date | why_it_matters

Checkpoint after EACH company so progress survives interruption. Report companies covered + how many CONFIRMED.
