# KARTO Atlas V2 — Implementation Instruction Set

**Audience:** a Claude Code session with full access to the repository.
**Your job:** read this document, inspect the actual codebase, then produce an implementation plan before writing code. Do not skip the plan step.
**Method:** every module below starts with WHY (the product reason), then WHAT (deliverable), then HOW (steps), then EXAMPLE (concrete, using the running example), then TEST (acceptance criteria).

---

## 0. Ground rules (read first)

**WHY:** V2 turns a descriptive census ("what exists") into a decision instrument ("so what, what next, for whom"). Every choice below serves that shift. These invariants protect the project's moat: auditability.

- **Source of truth stays `data/register.csv`.** Never hand-edit `data/atlas.json` — it is always regenerated by `scripts/cook_aggregates.py`.
- **Everything traceable.** Every derived number, label, or score must be reconstructible from register rows + documented rules. No black boxes.
- **Static-first.** The site is GitHub Pages. Every feature must work with build-time computation + client-side JS. No servers, no databases.
- **Rules over models.** Classifications are rule-based and transparent. LLMs may only be used as parsers (text → structured fields), never as data oracles at runtime.
- **Backward compatibility during transition.** When changing `atlas.json` shapes, keep old keys for one release with a deprecation note in the JSON meta.
- **Running example throughout:** BNP Paribas (FR, Financial Services) vs peers; Crédit Agricole as the silent company; Accor as non-bank cameo. All figures in examples are illustrative — verify against the real register.
- **Before planning:** inspect the actual repo. Confirm the real schema of `register.csv`, the current structure of `cook_aggregates.py` and `atlas.json`, and how the front-end loads data. Where this document conflicts with reality, flag it in your plan instead of guessing.

## PHASE 1 — Quick wins (no new data collection)

### Module B3 — Verdict split: 4 states instead of 3

**WHY:** The current "talk" verdict merges two totally different situations: *unconfirmed rumors* and *confirmed deployments that simply lack a value number*. A consultant reads these oppositely — one is noise, the other is a real program that hasn't published ROI (often a prime prospect). Merging them destroys the signal and mislabels honest activity as hype.

**WHAT:** Replace the 3-state cell verdict (`strong/active/talk`) with 4 states in `cook_aggregates.py` and the UI:
- `proven` — ≥40% of cell rows carry a value number
- `active` — ≥15% carry a value number
- `unquantified` — <15% with numbers BUT majority of rows are CONFIRMED
- `talk` — <15% with numbers AND majority of rows are CLAIMED/unconfirmed

**HOW:**
1. In the cook script, locate the verdict computation for `grid_global[]` and `grid_by_country{}` cells.
2. Add the existence-majority check to split the old `talk` bucket.
3. Keep the old `verdict` field one release; add `verdict_v2`. Document both in JSON meta.
4. Update UI legend, cell colors (add a 4th color distinct from pink), and the side-panel header badge.

**EXAMPLE:** France | Financial Services | Customer Support: 9 deployments, 8 CONFIRMED, 1 with a value number (11%). Today → "talk" (reads as vapor). After → `unquantified` (reads as: real activity, no published proof — a sales target, not a mirage). Meanwhile an Italian cell with 4 rows, all CLAIMED from press, 0 numbers → `talk` (correctly).

**TEST:** Recook; assert no cell with majority-CONFIRMED rows is labeled `talk`; assert legend shows 4 states; assert old field still present with deprecation note.

**WHY:** The site's brand IS credibility. Small unexplained oddities (density of 1.48×, stale rows, silent methodology) erode trust disproportionately. These are hours of work with outsized payoff.

## PHASE 2 — Universe & normalization

### Module A1 — Universe file

**WHY:** Two of the highest-value consultant questions are currently unanswerable: (1) "who was searched and has NOTHING" — absence is the money insight ("Bank B has no AI strategy"); (2) "is 6 deployments a lot for a company this size?" — raw counts without denominators compare elephants to mice. Both require knowing the full searched population with size data.

**WHAT:** `data/universe.csv` — one row per searched company (~1,964):
`company, country (2-letter CC), vertical, market_cap_usd, revenue_usd, employees, ticker`
Companies present in universe but absent from register = **L0 silent** (definition used by A4 and D4).

**HOW:**
1. Reconstruct the searched list from the original selection criteria (the register's own companies + whatever selection lists were used to pick the 1,964). If the original list isn't in the repo, flag this to the owner — do not invent it.
2. Enrich with size data from public sources (a one-time script; sources noted in a `universe_sources.md`).
3. Add integrity check to cook: every `register.csv` company must join to universe on (company, country). Emit `orphans.txt` for mismatches (spelling variants are the expected failure mode — build a small alias map).

**EXAMPLE:** `BNP Paribas, FR, Financial Services, 75000, 50000, 190000, BNP.PA`. Crédit Agricole appears in universe, has zero register rows → it now *exists* in the dataset as a silent company. Today it is literally invisible — the biggest insight the site cannot currently express.

**TEST:** Join coverage ≥98% (aliases resolve the rest); cook fails loudly on new orphans; `universe.csv` row count matches the documented searched population.

### Module B1 — Normalized metrics

**WHY:** "US has 501 deployments, France 91" is a population statement, not a performance statement. Consultants and investors need per-size intensity to say anything defensible.

**WHAT:** In cook, compute per company and per country: deployments per $B revenue, deployments per 10k employees, confirmed per company searched (existing density, now explained). Emit a new top-level `companies[]` array in `atlas.json`: `{slug, name, cc, vertical, deployments, confirmed, with_value_number, mktcap, revenue, employees, per_bn_revenue, per_10k_emp}`.

**EXAMPLE:** BNP: 6 deployments / $50B = 0.12 per $B. HSBC: 8 / $66B = 0.12. Raw counts said "HSBC leads"; normalized says "dead heat". That reversal is exactly the kind of claim a client pays for.

**TEST:** Spot-check 5 companies by hand; divisions guard against zero/missing denominators (emit null, never crash); `companies[]` present and documented in JSON meta.

---

### Module B2 — Peer benchmarks

**WHY:** A number without a reference class is decoration. "6 deployments" means nothing; "more than 88% of banks worldwide" is a slide. Percentile-in-peer-group is the atomic unit of benchmarking.

**WHAT:** For every company, compute percentile rank (0–100) within two peer groups — (a) same vertical globally, (b) same vertical × same country — on each of: deployment count, confirmed count, proof rate (share of rows with value number), and later maturity level (A4) and normalized intensity (B1). Store as `companies[].benchmarks: {global_vertical: {...}, country_vertical: {...}}`.

**HOW (caveman core):** take the peer list, sort by the metric, position of the company = percentile. Ties share rank. Peer groups smaller than 5 → emit null with reason (percentiles on 3 companies are noise).

**EXAMPLE:** 100 banks worldwide. Sorted by deployments, BNP beats 88 → 88th percentile. Sorted by proof rate, beats 60 → 60th. One-glance read: **does a lot, proves little** — which is simultaneously a strength claim and an engagement opportunity (someone should help them measure).

**TEST:** Percentiles in [0,100]; company at max metric = 100th; small peer groups null; BNP-style two-metric read reproducible from JSON alone.

### Module D4 — Silent companies view

**WHY:** This is the single highest-value consultant screen in V2: the prospect list. Every company here is a documented gap ("searched on DATE, zero disclosed AI") sitting next to its peers' activity — a cold-outreach machine with receipts.

**WHAT:** New route/view listing all L0 companies (in universe, absent from register). Columns: company, country, vertical, size (revenue or mktcap), peer context (median deployments of its vertical×country peers). Filters: country, vertical. Sort: size desc by default. Each row links to peers' evidence.

**EXAMPLE:** Filter France + Financial Services → Crédit Agricole tops the list (€2T assets, zero rows) with peer context "French banks median: 4 deployments". A consultant screenshots this single row into an outreach email. That email writes itself: "your five closest rivals disclose 23 deployments; you disclose zero — here are the 23 sources."

**TEST:** Every listed company verifiably absent from register; peer-context number recomputable; the France/banking example renders correctly against real data.

## PHASE 3 — Text extraction & hype detection

### Module A2 — Value extraction pass

**WHY:** Dollar figures are already sitting inside free-text `use_case` strings, invisible to computation. Extracting them turns anecdotes into a quantitative layer for free — no new collection.

**WHAT:** One-time (rerunnable) script: for each register row, parse `use_case` + `value_claimed` into `data/claims.csv`: `{row_id, amount, currency, claim_type: savings|revenue|investment|users|vague, period, source_phrase}`. `source_phrase` = the exact substring the extraction came from (auditability).

**HOW:**
1. First pass: regex for currency amounts (`$600M`, `€500M`, `¥2bn`, `1.2 billion`) and scale counts (`3M users`, `500 stores`).
2. Second pass: LLM classification of claim_type on rows where regex found an amount (batch, offline, results committed to the CSV — the site never calls an LLM).
3. Manual spot-check sample of 50 rows before merging.

**EXAMPLE:** Altria row: "AI part of $600M cost-savings initiative" → `{amount: 600, currency: USD, unit: M, claim_type: savings, source_phrase: "$600M cost-savings initiative"}`. A BNP-style row "expects €500M annual gains from GenAI by 2026" → `{500, EUR, M, savings, period: 2026}`.

**TEST:** Altria row extracts exactly as above; precision on the 50-row sample ≥90%; every claims row carries a `source_phrase` found verbatim in the register text.

### Module A5 — Row freshness

**WHY:** Intelligence has a shelf life. A 2023 deployment row read in 2026 may describe a dead program. Without freshness, the whole register silently rots and the first public embarrassment costs the credibility moat.

**WHAT:** Add `last_verified` (YYYY-MM) column to `register.csv` (backfill = original collection date). Cook computes staleness buckets: fresh (<12mo), aging (12–18), stale (>18). Emit per-row and aggregate staleness in `atlas.json`. UI shows badge (component from D8).

**EXAMPLE:** BNP's 2023 chatbot row: `last_verified: 2024-03` → in mid-2026 renders "⚠ verified 2024-03". Its 2025 GenAI row → "fresh". A consultant citing the page knows which claims to re-check before a client meeting — and the site saying so *increases* trust.

**TEST:** Column present on 100% of rows; buckets sum to row count; badge renders in side panel and (later) company pages.

### Module D6 — Hype detector

**WHY:** The site's founding thesis is "adoption everywhere, proof nowhere." This module *automates the thesis*: talk vs money, per company and per vertical. It is the most on-brand feature in V2 and highly shareable.

**WHAT:** Widget + view computing, per company and per vertical: announcements count (register rows), value-claims count (from A2), disclosed commitments (from A3 when available; until then, investment-type claims from A2). Display as a ratio/scatter: talk on one axis, substantiation on the other.

**EXAMPLE:** Bank X: 12 register rows, 0 value numbers, 0 investment figures → pure-talk quadrant. BNP: 6 rows, 1 value number, €100M disclosed investment → grounded quadrant. Energy vertical: 51 deployments, zero value numbers anywhere → the whole industry deploys blind, one chart.

**TEST:** Quadrant assignments recomputable from claims.csv + register; the energy-vertical read (if true in real data) renders as a single view; every plotted point links to its rows.

## PHASE 4 — Time dimension

### Module B4 — Momentum

**WHY:** `date` exists in the register and is currently dead weight. Time turns a snapshot into a story: who moved early, who is catching up, who went quiet. Investors and strategists think in trajectories, not levels.

**WHAT:** In cook, per company/vertical/country: deployments by year, YoY delta, `first_seen`, and a momentum label: `early_mover` (first_seen in the earliest tercile of its vertical) / `late_entrant` / `rising` (positive recent delta) / `flat` / `gone_quiet` (no rows in last 24mo despite history). Date conventions (document in JSON meta): `YYYY` → treated as mid-year; `YYYY-MM` → as-is; `missing` → excluded from time series AND counted in an `undated_pct` per entity (honesty metric).

**EXAMPLE:** BNP: first_seen 2021, +2 rows/year → `early_mover, rising`. Société Générale: first row 2025, three at once → `late_entrant, rising` — a budget just unlocked; a vendor reading this bids now. France banking overall: flat since 2023 while German banking accelerates post-2024 → a country-strategy slide in one chart.

**TEST:** Undated rows never appear in series but always in `undated_pct`; labels deterministic and documented; SocGen-style late-entrant pattern detectable in synthetic test data.

### Module D5 — Trends view

**WHY:** B4's arrays need a face. Time charts are the most instantly-read artifact in a client deck.

**WHAT:** View with: (1) deployments-over-time lines, selectable country/vertical overlays; (2) early-mover vs catch-up scatter (x = first_seen, y = recent momentum, size = deployments); (3) per-entity sparkline reused on company pages (D1).

**EXAMPLE:** Overlay France vs Germany, Financial Services: France plateaus 2023, Germany inflects 2024 → screenshot goes straight into a market-entry deck. On the scatter, SocGen sits far right (late) and high (rising) — the "catching up" quadrant is literally visible.

**TEST:** Charts render from `atlas.json` only (no runtime recomputation); undated share displayed as a caveat on every chart; sparkline component reusable.

## PHASE 5 — Maturity

### Module A4 — Maturity classifier (L0–L4)

**WHY:** One pilot ≠ an enterprise platform, and today the register can't tell them apart. True internal maturity is unknowable from outside — so we classify the **observable disclosure footprint** instead, with transparent rules. Humble, defensible, attackable-nowhere ("we label what they disclose, here are the rules").

**WHAT:** Script producing `data/maturity.csv`: `{company, cc, level, evidence_fields}` where `evidence_fields` lists exactly which rules fired.

Rule ladder (tune thresholds after inspecting real distributions — targets below are starting points):
- **L0 Silent** — in universe, zero register rows
- **L1 Talk** — rows exist, none CONFIRMED
- **L2 Pilot** — CONFIRMED rows, single horizontal, no value number, no named product
- **L3 Operating** — CONFIRMED and (≥2 horizontals OR named product OR scale numbers in text via A2)
- **L4 Industrialized** — L3 criteria AND multi-year presence (≥2 distinct years) AND (value number OR tier-P row)

Named-product detection: capitalized product token in `use_case` (e.g., "Romie", "BlueCruise", "AltriaGPT", "LUCIE") — regex + curated stoplist; store the matched name.

**EXAMPLE:** BNP: 6 rows, 4 horizontals, years 2023–2025, named product "LUCIE", 1 value number → L4, `evidence_fields: [multi_horizontal, named_product, multi_year, value_number]`. Crédit Agricole → L0. A single-row company with "piloting GenAI for internal docs", CONFIRMED, no number → L2.

**TEST:** Every level reconstructible from evidence_fields; distribution sanity (no level holds >60% of companies with rows — if it does, thresholds need tuning, flag it); the three example patterns classify as stated on synthetic rows.

**WHY:** The company is the natural unit of a consultant's question ("how is X doing?"), and today it doesn't exist as a page — company data is shredded across grid cells. This page is also the anchor for shareable URLs and the static API.

### Module B6 — Findings join

**WHY:** `findings.csv` (regulator warnings, job cuts, skeptic press) exists and is invisible. It is the counterweight that keeps the site honest — deployments next to warnings, not instead of them.

## PHASE 6 — Query, compare, personas

### Module C1 — Compare (client-side function)

**WHY:** "Bank A vs Bank B" is the original consultant question and still requires ~40 clicks and a notepad. Comparison must be one action. Static site → this is a JS function over already-loaded JSON, not a server endpoint.

**WHAT:** JS module: input = 2–5 entity refs (companies OR countries OR verticals — homogeneous per call); output = aligned metric table: deployments, confirmed %, proof %, maturity (companies), momentum label, normalized intensity, commitments total, benchmark percentiles. One output shape for all three entity types (nulls where a metric doesn't apply).

**EXAMPLE:** `compare(['bnp-paribas','hsbc','santander'])` → `{metrics: [{key:'deployments', values:[6,8,4]}, {key:'maturity', values:['L4','L4','L2']}, {key:'proof_rate', values:[0.17,0.25,0]}, {key:'commitments_usd', values:[108,1000,null]}, ...]}`.

**TEST:** Pure function, no fetch beyond initial JSON; mixed entity types rejected with clear error; output renders D2 without transformation.

### Module C2 — Filter grammar + 3-tier NL stack

**WHY:** "EU banks with no customer-support AI" is one thought; it must be one action. But LLM answers over data can hallucinate — so the LLM's only job is translating language into a deterministic filter that JS executes. Trust through transparency: the parsed filter is always shown back as editable chips.

**WHAT:**
1. **Grammar (JSON schema, published in repo):** `{country: [CC], region: [EU|APAC|...], vertical: [], horizontal: [], maturity_min: L0-L4, has_value_number: bool, existence: [], tier: [], year_range: [from,to], silent_only: bool, text: string}`. Regions expand to country lists at parse time.
2. **Tier 1 — Chips UI:** every grammar field as clickable facets. Always available, zero-LLM, and the render target for tiers 2–3.
3. **Tier 2 — Dictionary mapper (~100 lines):** synonym tables ("banks"→Financial Services; "EU"→[FR,DE,IT,ES,...]; "no|without|zero"→silent_only or has_value_number:false by context). Catches ~70% of typed queries offline.
4. **Tier 3 — Light LLM (optional, feature-flagged):** Haiku-class, JSON-mode, prompt = grammar schema + 5 worked examples. Handles odd phrasing and out-of-scope explanations. Requires either user-supplied API key or a minimal proxy — never ship the owner's key in client JS. Log unparseable queries (client-side list the owner can export) — they are the grammar's roadmap.

**EXAMPLE:** Type "EU banks with no customer-support AI" → tier 2 resolves fully → chips: [region:EU] [vertical:Financial Services] [horizontal:Customer Support] [silent_only ✓] → result: Crédit Agricole, Commerzbank, ... Type "Malaysian Police AI" → tier 2 resolves [country:MY], fails on "Police" → tier 3 replies: "out of scope — dataset covers listed companies; closest: MY + Security/Risk function?" and proposes those chips.

**TEST:** The EU-banks query returns the correct list on real data via tier 2 alone (no LLM); every executed filter visible as chips and hand-editable; LLM path emits only schema-valid JSON (invalid → fall back to chips with error note).

## PHASE 7 — Push layer, scores, API, export

### Module B7 — Insight generator (push layer) + actionability rule

**WHY:** Everything above is pull — it answers questions the user knew to ask. Intelligence also means the data speaking first. And a hard product rule applies: **no insight ships without an action** — the QA test is "would a consultant paste this into a client email?" If no, cut it.

**WHAT:** Cook-time rule engine producing `atlas.json → insights[]`, each card:
`{id, type, finding, meaning, action, persona, ammo_urls[], entities[], surprise_score}`.
Rule types for v1 (one action-template per type, reused across all matches):
1. **Outlier** — entity ≥2 z-scores from its peer group on any metric → action: "benchmark deck" (consultant) / "position review" (investor)
2. **Contradiction** — high talk, zero substantiation (needs A2): → action: "pre-written earnings-call question" / "discount in DD"
3. **Whitespace** — empty cell where peer countries are active → action: "greenfield pitch list + foreign proof-cases as references" (vendor/consultant)
4. **Momentum break** — late entrant surging or veteran gone quiet (needs B4) → action: "bid window open" / "check before the board asks"
5. **Silent giant** — top-decile size, L0 (needs A1) → action: "cold-outreach with peer-gap ammo"
Feed view: cards sorted by surprise_score, filterable by persona.

**EXAMPLE:** Silent-giant rule fires on Crédit Agricole → card: finding "€2T-asset bank, zero disclosed AI, French bank median: 4" / action "outreach opener: your 5 closest rivals disclose 23 deployments (sources attached)" / persona: consultant, vendor / ammo: the 23 source URLs / button: **Export prospect brief**.

**TEST:** Every card has non-empty action + ammo_urls; rules deterministic (same data → same cards); each card's button leads somewhere real (export, list, view).

### Module B5 — Opportunity scores

**WHY:** D4 and B7 surface opportunities qualitatively; scores let views *sort* by opportunity. Formulas must be simple and published — a score nobody can recompute is a liability for this brand.

**WHAT:** Cook computes, with formulas documented in JSON meta:
- `whitespace_score` per grid cell: f(peer-country activity in same vertical×horizontal, own-cell emptiness, market size)
- `prospect_score` per company: f(confirmed activity, absence of value numbers, absence of named vendor, size)
Both 0–100, monotonic, no ML.

**EXAMPLE:** France|Hospitality|Security-Risk: empty, while UK+US hospitality run 9 deployments there → whitespace 85 → sorts to top of the consultant tile; clicking lists French hotel groups (Accor first, by size) with the 9 foreign proof-cases attached as references.

**TEST:** Formulas in meta; recomputable by hand for any cell/company; sorting integrated in D3/D4 views.

### Module A3 — Commitments file

**WHY:** `value_claimed` is output-talk; money committed is input-proof. Input vs output direction is itself a signal: input with no output after 2 years = red flag; output with no input = suspicious claim. This completes the hype detector (D6) with its money axis.

**WHAT:** `data/commitments.csv`: `{company, cc, amount, currency, commitment_type: invest|acquire|partner|savings_claim|revenue_claim, horizon: one_off|multi_year, tier: P|I, source_url, date}`. Pilot scope: ONE vertical (Financial Services) collected properly, before scaling. Amounts always accompanied by source; undisclosed-amount commitments (partnerships) are valid rows with `amount: null`.

**EXAMPLE:** BNP: row 1 `{partner, Mistral AI, amount: null, tier: P, <press release URL>}`; row 2 `{invest, 100M EUR, multi_year, tier: P, <annual report URL>}`. Now the company page shows: money in €100M / claimed out €500M — and an analyst has an earnings-call question for free.

**TEST:** Schema validation in cook; banks pilot populated; D6 and D1 consume it without code changes (fields designed now).

### Module C3 — Static API

**WHY:** Machines are users: notebooks, Excel, AI agents. Today the data exists only rendered inside JS — (proven painfully: an AI assistant trying to read this site's data had no path to it). Flat per-entity JSON = stable, citable, zero-cost distribution. Vacuumability is accepted: the data already ships in `atlas.json`; the moat is methodology + update heartbeat, not the snapshot. Optionally add a LICENSE (CC BY-NC) and keep raw `register.csv` out of the published set.

**WHAT:** Cook additionally writes:
- `api/company/{slug}.json` — the D1 payload per company
- `api/country/{cc}.json` — country rollup + grid
- `api/vertical/{slug}.json` — vertical rollup
- `api/index.json` — catalog of all entities + schema version + license
All plain files served by GitHub Pages.

**EXAMPLE:** `GET https://<site>/api/company/bnp-paribas.json` from a Python notebook → 5KB of everything on BNP, sources included. A consultant's report cites that URL as its data source.

**TEST:** Files regenerate on every cook; index lists 100% of entities; a fresh curl of the BNP URL returns valid JSON matching the company page.

### Module D7 — Report export

**WHY:** Consultants live in decks and briefs. A one-click briefing converts site value into work-product value — the moment KARTO enters a client deliverable, it starts building the trust brand.

**WHAT:** "Generate briefing" button on: country view, vertical view, company page (D1), compare view (D2), insight cards (B7). Output: markdown (v1; PDF later via print stylesheet) containing: title + date + data version, key stats, the relevant charts (as embedded SVG or data tables), insight cards in Finding→Action format, and a complete numbered source list.

**EXAMPLE:** On the BNP-vs-rivals compare view → click → `briefing-bnp-hsbc-santander-2026-07.md`: the metric table, maturity levels with rule evidence, commitments, ACPR flag, and 14 numbered source URLs — the appendix of a client deck, produced in 10 seconds.

**TEST:** Every number in the export traceable to a listed source or documented rule; markdown renders cleanly in standard viewers; data version stamped.

## Build order & dependencies (implement in this sequence)

```
Phase 1: B3 → D8                      (no new data; ship fast)
Phase 2: A1 → B1 → B2 → D4            (universe unlocks everything)
Phase 3: A2 → A5 → D6                 (extraction; hype detector)
Phase 4: B4 → D5                      (time)
Phase 5: A4 → B6 → D1                 (maturity; company pages)
Phase 6: C1 → C2 → D2 → D3            (query & personas)
Phase 7: B7 → B5 → A3 → C3 → D7       (push, scores, money, API, export)
```
